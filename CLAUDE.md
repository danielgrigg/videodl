## What videodl is

A single-container, server-rendered web app for downloading source videos with yt-dlp and browsing/playing them on a home Ubuntu server, reached over Tailscale at `video.griggx.com`. No auth (tailnet is the perimeter), no database (filesystem is the source of truth; in-memory job registry covers in-flight state only).

## Commands (once scaffolded)

- Dev server: `uv run uvicorn app.main:app --reload`
- Typecheck: `uv run mypy app` — must pass under `--strict` (already configured in `pyproject.toml`)
- Add a dependency: `uv add <pkg>` (commit the resulting `uv.lock`)
- Refresh yt-dlp (extractors rot fast): `uv lock --upgrade-package yt-dlp` then rebuild the image
- Container: `docker compose up --build`

## Architecture invariants

These are decisions from the plan that are easy to violate accidentally — treat them as load-bearing:

- **yt-dlp is used as a library, not shelled out.** Progress hooks push state into the typed `Job` model; do not call the CLI.
- **`Any` is quarantined to one boundary.** yt-dlp ships no type stubs, so the progress-hook payload (`dict[str, Any]`) is the *only* place `Any` is tolerated. Convert to the typed `Job` immediately; don't let `Any` propagate. `mypy --strict` enforces this everywhere else.
- **Job registry is in-memory; filesystem is the source of truth.** Completed downloads survive restarts because they're just files in `VIDEO_DIR`. Losing in-flight jobs on restart is acceptable — do not add a database to "fix" this.
- **Bounded concurrency.** yt-dlp blocks; it runs on a `ThreadPoolExecutor(max_workers=settings.max_concurrent)` off the event loop. A pasted playlist must not saturate the box.
- **Live progress flows via SSE** (`sse-starlette`'s `EventSourceResponse`) reading from the typed job dict. `Job` is a pydantic model so `.model_dump()` is the serialisation path — no hand-written glue.
- **Range requests come for free** from Starlette's `FileResponse`; do not write custom byte-range handling for `/videos/{name}`.
- **Thumbnails** live in `.thumbs/` *inside* `VIDEO_DIR` (sibling to videos). Generated by ffmpeg at ~10% into the file, on download-finish, plus a startup sweep to backfill existing videos.
- **Remux only, no transcode.** `merge_output_format=mp4` is cheap; the known caveat is VP9/AV1 sources won't play in Safari. Only revisit with transcode-on-demand if Safari/iOS becomes a requirement.

## Config

Typed via `pydantic-settings` in `app/config.py`. Env vars: `VIDEO_DIR` (default `/data/videos`), `MAX_CONCURRENT` (default `2`). The container mounts `/srv/data0/studio/videos` → `/data/videos`; the process must have read-write on the host path (match UID/GID or `chown`).
